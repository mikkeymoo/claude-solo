# /sql-dev Command

Comprehensive SQL development command for database design, migration, optimization, and testing.

## Usage
```
/sql-dev [action] [options]
```

## Actions

### init - Initialize Database Project
```
/sql-dev init [project-name] [--engine postgres|mysql|mssql|sqlite]
```

Creates database project structure:
```
database/
├── migrations/
│   └── 001_initial_schema.sql
├── seeds/
│   └── 001_test_data.sql
├── procedures/
├── functions/
├── views/
├── triggers/
├── indexes/
├── tests/
├── backups/
├── config/
│   └── database.yml
└── README.md
```

### schema - Design Database Schema
```
/sql-dev schema [--visualize] [--normalize]
```

Features:
- Interactive table designer
- Relationship mapping
- Normalization analysis (1NF, 2NF, 3NF, BCNF)
- ERD generation
- Schema validation

### migrate - Database Migrations
```
/sql-dev migrate [up|down|status] [--version]
```

Migration management:
- Version control for schema changes
- Rollback capabilities
- Migration history tracking
- Conflict detection
- Dry-run mode

### optimize - Query Optimization
```
/sql-dev optimize [--query "SELECT..."] [--explain]
```

Optimization features:
- Query plan analysis
- Index recommendations
- Statistics updates
- Query rewriting suggestions
- Performance benchmarking

### test - Database Testing
```
/sql-dev test [--fixtures] [--coverage]
```

Testing capabilities:
- Unit tests for procedures/functions
- Data integrity tests
- Performance tests
- Regression tests
- Test data generation

### backup - Backup Management
```
/sql-dev backup [--full|--incremental] [--compress]
```

Backup operations:
- Full database backup
- Incremental backups
- Point-in-time recovery
- Backup verification
- Automated scheduling

### analyze - Performance Analysis
```
/sql-dev analyze [--slow-queries] [--indexes] [--locks]
```

Analysis tools:
- Slow query log analysis
- Missing index detection
- Lock wait analysis
- Table bloat detection
- Connection pool monitoring

## Schema Design Patterns

### Normalized Schema
```sql
-- Generated by /sql-dev schema --normalize

-- Users table (3NF)
CREATE TABLE users (
    user_id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- User profiles (separated for normalization)
CREATE TABLE user_profiles (
    profile_id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(user_id) ON DELETE CASCADE,
    first_name VARCHAR(100),
    last_name VARCHAR(100),
    bio TEXT,
    avatar_url VARCHAR(500)
);

-- Addresses (reusable component)
CREATE TABLE addresses (
    address_id BIGSERIAL PRIMARY KEY,
    street VARCHAR(255),
    city VARCHAR(100),
    state VARCHAR(50),
    country VARCHAR(50),
    postal_code VARCHAR(20)
);

-- User addresses (many-to-many)
CREATE TABLE user_addresses (
    user_id BIGINT REFERENCES users(user_id),
    address_id BIGINT REFERENCES addresses(address_id),
    address_type VARCHAR(20),
    is_primary BOOLEAN DEFAULT false,
    PRIMARY KEY (user_id, address_id)
);
```

### Audit Pattern
```sql
-- Generated by /sql-dev schema --audit

CREATE TABLE audit_log (
    audit_id BIGSERIAL PRIMARY KEY,
    table_name VARCHAR(100) NOT NULL,
    operation VARCHAR(10) NOT NULL,
    user_id BIGINT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    old_data JSONB,
    new_data JSONB,
    ip_address INET,
    user_agent TEXT
);

-- Auto-generate audit triggers
CREATE OR REPLACE FUNCTION audit_trigger_function()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO audit_log (table_name, operation, user_id, old_data, new_data)
    VALUES (TG_TABLE_NAME, TG_OP, current_user_id(),
            row_to_json(OLD), row_to_json(NEW));
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## Query Optimization Examples

### Before Optimization
```sql
-- Identified by /sql-dev optimize
SELECT u.*, p.*, COUNT(o.order_id) as order_count
FROM users u
LEFT JOIN profiles p ON u.user_id = p.user_id
LEFT JOIN orders o ON u.user_id = o.user_id
WHERE u.created_at > '2024-01-01'
GROUP BY u.user_id, p.profile_id;
```

### After Optimization
```sql
-- Optimized version
WITH user_orders AS (
    SELECT user_id, COUNT(*) as order_count
    FROM orders
    WHERE user_id IN (
        SELECT user_id FROM users
        WHERE created_at > '2024-01-01'
    )
    GROUP BY user_id
)
SELECT u.*, p.*, COALESCE(uo.order_count, 0) as order_count
FROM users u
LEFT JOIN profiles p ON u.user_id = p.user_id
LEFT JOIN user_orders uo ON u.user_id = uo.user_id
WHERE u.created_at > '2024-01-01';

-- Recommended indexes
CREATE INDEX idx_users_created_at ON users(created_at);
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

## Testing Framework

### Test Structure
```sql
-- tests/test_user_functions.sql
BEGIN;

-- Test user creation
SELECT plan(5);

-- Test 1: Create user successfully
SELECT ok(
    create_user('testuser', 'test@example.com'),
    'User creation should succeed'
);

-- Test 2: Duplicate username fails
SELECT throws_ok(
    'SELECT create_user(''testuser'', ''other@example.com'')',
    '23505',
    'Duplicate username should fail'
);

-- Test 3: Invalid email fails
SELECT throws_ok(
    'SELECT create_user(''newuser'', ''invalid-email'')',
    '23514',
    'Invalid email should fail'
);

-- Finish tests
SELECT * FROM finish();
ROLLBACK;
```

## Performance Monitoring

### Query Performance Dashboard
```sql
-- Generated by /sql-dev analyze --slow-queries

-- Top 10 slowest queries
SELECT
    query,
    calls,
    mean_exec_time,
    total_exec_time,
    min_exec_time,
    max_exec_time,
    stddev_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- Missing indexes
SELECT
    schemaname,
    tablename,
    attname,
    n_tup_ins + n_tup_upd + n_tup_del as write_activity,
    seq_scan * seq_tup_read as read_activity
FROM pg_stat_user_tables
JOIN pg_attribute ON (attrelid = (schemaname||'.'||tablename)::regclass)
WHERE seq_scan > idx_scan
ORDER BY read_activity DESC;
```

## Best Practices Enforced

1. **Always use transactions** for data modifications
2. **Parameterized queries** to prevent SQL injection
3. **Proper indexing** on foreign keys and WHERE clauses
4. **Consistent naming** conventions
5. **Audit trails** for sensitive data
6. **Regular backups** with verification
7. **Query optimization** before production
8. **Test coverage** for all procedures/functions
